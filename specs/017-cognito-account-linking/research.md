# Research: Cross-Provider Cognito Account Linking

## Decision: SST v4's `triggers.preSignUp` is a real, working feature (unlike `addIdentityProvider()`)

**Rationale**: PLAN.md flagged a specific risk — SST v4's `pool.addIdentityProvider()` is confirmed broken in this codebase (`aws-infra/AGENTS.md`, "Known SST v4 bugs"; `app-stack.ts` already works around it by constructing `aws.cognito.IdentityProvider` directly from `@pulumi/aws`). Reading `aws-infra/.sst/platform/src/components/aws/cognito-user-pool.ts` directly (the actual Pulumi component source SST generates locally, not just its `.d.ts` surface) shows `CognitoUserPoolArgs.triggers` is a distinct, independently-implemented code path: `normalizeTriggers()` builds a `Function` (or accepts a bare ARN) per trigger key including `preSignUp`, and wires it via `cognito.UserPool`'s native `lambdaConfig` argument — the same mechanism CloudFormation/Pulumi's AWS provider uses natively, not a custom SST reconciliation loop like the identity-provider bug was. There is no equivalent codepath, so the two bugs are unrelated, but this is still confirmed against a real dev deploy in Phase 1 (implementation) rather than assumed from reading source alone, per PLAN.md's explicit caution.

**Alternatives considered**: Fall back to raw `aws.cognito.UserPool` + `aws.lambda.Permission` (the same "escape hatch" pattern used for `googleIdP`/`facebookIdP`) if the live deploy shows `triggers.preSignUp` doesn't actually attach. Rejected as the default plan since the source inspection shows a legitimately different, standard implementation; only fall back if live verification proves it broken.

## Decision: Lambda receives `PreSignUp_ExternalProvider` as one of several trigger sources on the same `preSignUp` hook

**Rationale**: Cognito invokes the single configured `preSignUp` Lambda for multiple trigger sources: `PreSignUp_SignUp` (native email/password), `PreSignUp_ExternalProvider` (federated), `PreSignUp_AdminCreateUser`. The Lambda MUST branch on `event.triggerSource` and pass the event through unchanged for anything other than `PreSignUp_ExternalProvider` — a native email/password sign-up is not a linking candidate (it either has no external identity to link, or in the case of "this native user IS the intended future canonical account," no action is needed).

**Alternatives considered**: A separate trigger just for external providers. Rejected — Cognito does not expose a way to register a per-source Lambda; a single `preSignUp` slot handles all sources, and the discrimination happens inside the function.

## Decision: Facebook has no `email_verified`-equivalent attribute to map; treat Facebook emails as verified by convention

**Rationale**: Google is an OIDC IdP and exposes `email_verified` as a literal claim, directly mappable via `attributeMapping: { email_verified: "email_verified", ... }` on the existing `aws.cognito.IdentityProvider` resource (`app-stack.ts:62-66`). Facebook's Cognito IdP integration is backed by the Graph API, which has no equivalent field — Facebook does not expose whether an email was verified because Facebook itself only ever grants the `email` OAuth permission for addresses it has already verified server-side (documented Facebook Login platform policy). There is nothing to map from Facebook's attribute schema. The Lambda therefore hardcodes `verified = true` whenever `event.userName` (or the incoming provider name) indicates the Facebook IdP, and reads the mapped `email_verified` user attribute for Google. This assumption is documented in `aws-infra/AGENTS.md`.

**Alternatives considered**: Refuse to ever link Facebook accounts (safest, but defeats the point of Item 16 for the exact IdP that motivated it — bruno's Facebook duplicate). Rejected as it leaves the feature not covering its primary driving case.

## Decision: matchable existing users are `CONFIRMED` or `EXTERNAL_PROVIDER`, not `CONFIRMED` alone

**Rationale**: Confirmed live against the dev pool during T008 (`aws cognito-idp list-users`) — a pre-existing federated shadow user (e.g. bruno's Google or Facebook account) has `UserStatus: EXTERNAL_PROVIDER`, never `CONFIRMED`. This is Cognito's permanent steady-state for any user that only ever authenticated via a federated identity; it does not transition to `CONFIRMED` the way a native account does after email verification. The original design (and initial spec wording) assumed `CONFIRMED` alone, which would have silently excluded every already-federated account from ever being matched — breaking User Story 1's own three-way-linking scenario (Facebook linking onto an account that already has Google linked). The `ListUsers` filter and the migration script's grouping both match on `CONFIRMED` OR `EXTERNAL_PROVIDER`, excluding other statuses (`UNCONFIRMED`, `RESET_REQUIRED`, `FORCE_CHANGE_PASSWORD`, `COMPROMISED`, `ARCHIVED`) that represent incomplete or problematic accounts unsafe to link against.

**Alternatives considered**: Matching on `CONFIRMED` only (the original assumption) — rejected once live testing showed it silently breaks the feature's own P1 scenario.

## Decision: `AdminLinkProviderForUser` requires `ProviderAttributeValue` (the external subject id), sourced from the event, not looked up separately

**Rationale**: The Pre Sign-up event for `PreSignUp_ExternalProvider` carries `event.userName` in the form `<provider>_<subject>`, lowercase provider prefix (e.g. `facebook_122094143511399836`, matching the existing dev duplicate accounts' observed usernames in PLAN.md Item 16's "Current state"). The subject portion after the provider prefix is exactly the `ProviderAttributeValue` needed for `AdminLinkProviderForUser`'s `SourceUser.ProviderAttributeValue` (with `ProviderAttributeName: "Cognito_Subject"`). No additional external API call is needed to obtain it.

**Alternatives considered**: Deriving the subject from `event.request.userAttributes.sub` — rejected, `sub` at this point in the lifecycle refers to the *not-yet-created* shadow user, not the external identity's own subject id, which is the wrong value for `SourceUser`.

## Decision: the migration script cannot check `ops` emptiness directly — that check is a manual operator pre-step, not automated

**Rationale**: Discovered while implementing T014 — `aws-infra/stacks/platform.ts` puts Aurora Postgres in a private VPC subnet, with its security group (`DbSg`) allowing inbound Postgres traffic only from the Lambda security group (`lambdaSg`), no public accessibility, no Data API. A script run locally (`npx tsx scripts/merge-duplicate-cognito-users.ts`) has no network path to that database at all — not a credentials problem, a routing one; no bastion, VPN, or SSM tunnel exists in this infra today. The original plan (`hasWalletData` via a direct `SELECT 1 FROM ops ...`) is not achievable from a local script without adding new networking infrastructure, which is out of scope for this feature (see Assumptions — no `crypto-assist` code changes, and no new infra beyond the Lambda trigger itself). Instead, the script identifies duplicate groups and their `sub`/creation-date/username via Cognito only, and prints them clearly in `--dry-run` output; the operator manually checks each account's wallet (e.g. by signing into the app with each identity, or via whatever direct DB access they already have set up) *before* running the script for real, per the updated `quickstart.md`. The "leave orphaned rather than auto-merge" safety property from FR-006/User Story 4 is preserved either way — the script never touches `ops` regardless — this only changes who performs the emptiness check, not what happens as a result of it.

**Alternatives considered**: Adding a new backend API endpoint to expose per-user `ops` emptiness — rejected, out of scope (this feature is `aws-infra`-only per its own Assumptions, and a new authenticated admin endpoint is a meaningfully sized addition of its own). Setting up a bastion/SSM tunnel to reach Aurora from a local script — rejected as infrastructure scope creep for a single one-off script; worth reconsidering only if this kind of one-off migration becomes a recurring need.

## Decision: Retroactive migration of pre-existing duplicates is a standalone script, not part of the Pulumi stack

**Rationale**: Per clarification, User Story 4's merge of pre-existing duplicates (bruno's 3 dev accounts) is an operator-run, one-off action, not a continuously-reconciled infrastructure resource — it does not belong in `app-stack.ts` (which SST/Pulumi treats as declarative desired state, re-applied on every deploy; a "merge these specific accounts" action is inherently imperative and one-shot, and would either no-op oddly on every future deploy or need brittle state-tracking to avoid re-running). It is implemented as `aws-infra/scripts/merge-duplicate-cognito-users.ts`, invoked manually (e.g. `npx tsx scripts/merge-duplicate-cognito-users.ts --stage dev`), using `@aws-sdk/client-cognito-identity-provider` directly (no Pulumi/SST involvement) against a target user pool ID (read from `/{app}/{stage}/CognitoUserPoolId` SSM, matching the pattern already established for reading other cross-stack SSM outputs).

**Alternatives considered**: A Lambda invoked manually via `aws lambda invoke`. Rejected — a local Node script is simpler to run, iterate on, and read output from during a manual one-off dev-environment operation; no deployment step needed to change its logic between attempts.

## Decision (superseded — see below): non-canonical duplicate accounts were originally planned to be disabled after linking, not deleted

**Original rationale** (kept for history): `AdminLinkProviderForUser` was planned to repoint the external identity's federation mapping to the canonical (oldest) user, after which `AdminDisableUser` on the now-redundant duplicate would prevent any lingering direct sign-in path, while remaining reversible. This turned out not to be how the API actually behaves — see the superseding decision immediately below, discovered while executing T016 live.

## Decision: `AdminLinkProviderForUser` cannot merge already-existing accounts at all — non-canonical federated duplicates are deleted, not linked or disabled

**Rationale**: Discovered live during T016 — calling `AdminLinkProviderForUser` against bruno's real pre-existing Google/Facebook duplicate accounts failed with `InvalidParameterException: Merging is not currently supported, provide a SourceUser that has not been signed up in order to link`. This is a hard, documented AWS Cognito API constraint, not a bug in this feature's code: the API only links a federated identity that has **not yet** created its own Cognito shadow user — exactly the Pre Sign-up Lambda's use case (User Stories 1–3), which intercepts *before* that shadow user exists. Once a federated identity already has its own independent, already-signed-up user record (exactly what "pre-existing duplicate" means), there is no supported Cognito API to retroactively merge it into a different destination account.

The only real fix: delete (`AdminDeleteUser`) the non-canonical federated duplicate's Cognito record entirely, then have that person sign in again via the same provider. That second sign-in is now a genuinely new federated sign-up (no existing shadow user for that identity anymore), which the already-deployed and already-verified Pre Sign-up Lambda correctly intercepts and links to the canonical account — the exact working mechanism from User Story 1. This changes FR-006/User Story 4 from "link, then disable" to "delete, then the person re-authenticates" — a two-step process (one automated, one requiring the person's own action) rather than a single silent script run. A non-canonical **native** (email/password) duplicate has no provider to re-authenticate through, so it is disabled (not deleted) as originally planned — it simply can't be resolved by this mechanism at all, only prevented from being used again.

Idempotency (User Story 4's acceptance scenario 4/5) is preserved, and more simply than originally designed: a deleted user vanishes from `ListUsers` entirely, so a re-run of the script naturally finds no duplicate group for that email and reports it as already resolved — no explicit "already processed" tracking needed.

**Alternatives considered**: Keeping `AdminLinkProviderForUser` as the mechanism and treating the "already signed up" error as itself the desired no-op outcome — rejected; the error means the link was never established at all (not "already linked, safely skip"), so treating it as success would silently leave the accounts unmerged while claiming success. Building a data-migration path that reassigns the `ops` table's `user_id` from the non-canonical `sub` to the canonical `sub` directly — rejected as out of scope (this feature explicitly leaves wallet data alone per FR-006, and reassigning ownership of financial records via a side-channel script bypasses the backend's own data model entirely).

## Decision: No new automated test framework for `aws-infra`; use Node's built-in `node:test` + `node:assert`

**Rationale**: Per clarification, only the new Lambda's pure decision logic needs a unit test, scoped narrowly. `aws-infra` has zero existing test infrastructure (confirmed via `package.json` — no `vitest`/`jest`/test script) and the constitution's "No Speculative Code" / dependency principle requires checking whether functionality can be achieved without a new package first. Node 20+ (the Lambda runtime already targeted) ships `node:test` and `node:assert/strict` with no installation step, runnable via `node --test`. This avoids introducing a full test framework (Vitest, Jest) into a repo that has consciously avoided one, while still satisfying the narrow, function-scoped testing the clarification asked for.

**Alternatives considered**: Vitest (already used in `crypto-assist/web`) — rejected for `aws-infra` specifically since it would be a net-new dependency solely to test one function, contradicting the "don't add a package that would be the sole consumer of it" guidance; `node:test` has zero installation cost.

## Decision: AWS SDK calls isolated behind a thin interface for testability (FR-008)

**Rationale**: The decision logic (given an email + verified flag + trigger source, decide: link / pass-through / fail-closed) must be unit-testable without live AWS calls. The handler is split into: (a) a pure `decide(...)` function taking already-fetched data (matching users, verified flag) and returning a decision, unit-tested directly with `node:test`; and (b) the Lambda entrypoint (`index.ts`) that calls the real `@aws-sdk/client-cognito-identity-provider` `ListUsersCommand`/`AdminLinkProviderForUserCommand` and feeds their results into `decide(...)`. Only (a) needs a test double; (b) is exercised by live dev deploy verification, consistent with the rest of `aws-infra`.

**Alternatives considered**: Mocking the AWS SDK client itself (e.g. `aws-sdk-client-mock`) — rejected as a new dependency; a plain function boundary needs no mocking library at all.

## Decision: a retry of an already-linked identity is caught and treated as a no-op

**Rationale**: Confirmed live during T008 via synthetic `aws lambda invoke` testing against a disposable test user — re-invoking the trigger for an identity already linked in a prior attempt (simulating a client retry after e.g. a network blip) throws `InvalidParameterException: SourceUser is already linked to DestinationUser` from `AdminLinkProviderForUser`. The original implementation let this propagate as an unhandled error, which would fail-close a legitimate retry — violating the spec's own stated edge case ("the retry must not attempt to link the same identity twice or error the second time"). Fixed by catching specifically `InvalidParameterException` with an "already linked" message and treating it as success (still sets `autoConfirmUser`/`autoVerifyEmail`); any other error from that call still fails closed per FR-005 unchanged.

**Alternatives considered**: Checking whether the identity is already linked before calling `AdminLinkProviderForUser` (e.g. via `AdminGetUser` or inspecting `ListUsers` results for an existing alias) — rejected as an unnecessary extra API call; catching the specific, well-documented error Cognito already returns for exactly this case is simpler and just as correct.

## Decision: Ambiguous match (>1 existing user with the same verified email) fails closed, same as an SDK error

**Rationale**: FR-003/FR-005 both resolve to the same fail-closed policy per clarification. Treating "more than one match" identically to "an unexpected error" keeps the decision function's contract simple: it has exactly three outcomes (`link`, `pass-through`, `fail`), not four, and avoids a subtly different code path for an edge case that is itself an anomaly (multiple existing users sharing a verified email should not normally occur once this feature is live — it can only arise from data created before this feature, i.e. exactly what User Story 4's migration cleans up).

**Alternatives considered**: A distinct "ambiguous" outcome separate from "error" — rejected, no behavioral difference is needed (both must fail closed), so the added enum variant would be speculative.
